## 1. 项目概述

### 1.1 产品定位

AI虚拟恋爱聊天产品：用户选择一个有人设的虚拟男友角色，通过文字聊天互动。他会回复文字、发语音消息、还会主动发"自拍照片"。

### 1.2 目标用户

| 属性 | 描述 |
|------|------|
| 性别 | 女性为主 |
| 年龄 | 16-25岁 |
| 场景 | 恋爱体验、情感陪伴、闲聊娱乐 |
| 需求 | 追求真实感、有趣的互动、情感共鸣 |

### 1.3 核心痛点

市面上现有AI聊天产品的三大问题：
- **回复模板化**：AI味太重，一眼假
- **语气太正式**：不像真人聊天，缺乏网络用语的自然感
- **缺乏记忆和情感共鸣**：聊完就忘，不像"真人"

### 1.4 核心特性

| 特性 | 说明 |
|------|------|
| 角色扮演对话 | LLM扮演有完整人设的虚拟男友，性格鲜明、会撩、有记忆 |
| 语音消息 | TTS自动生成语音，不同角色不同声线 |
| AI自拍照片 | 对话中角色会主动"发照片"（AI实时生成） |
| 微信聊天界面 | 像跟真人聊微信一样的体验 |
| 短期记忆 | 单次会话内记住之前聊过的内容 |
| 状态系统 | 虚拟男友有自己的日程/状态，影响对话语气 |

### 1.5 产品流程

```
选择角色 → 进入聊天界面 → 自由聊天（无轮次限制）→ 角色回复文字+语音+偶尔发照片
```

### 1.6 与前两个产品的对比

| | 黑话翻译器 | 哄哄模拟器 | 纸片人男友 |
|---|---|---|---|
| **积木** | 1块（LLM） | 2块（LLM+TTS） | 3块（LLM+TTS+图像生成） |
| **交互** | 输入→看 | 选择→看+听 | 聊天→看+听+看图 |
| **形态** | 工具 | 游戏 | 陪伴 |

### 2.2 功能详细说明

#### F01 角色选择

**功能描述**：用户进入产品后，从4个预设角色中选择一个虚拟男友作为聊天对象。

**业务规则**：
- 每个角色拥有不同的**性格标签**和**外貌描述**
- 角色选定后**不可切换**
- 用户刷新页面后，角色保持不变（单次会话内）

**预设角色列表**：

| ID | 名字 | 一句话介绍 | 性格标签 | 说话风格 | 声音 |
|---|---|---|---|---|---|
| warm-boy | 林屿 | 大学同班同学，温柔学长型 | 温柔、体贴、细心 | 说话轻柔，网络用语+emoji+偶尔撒娇，常用"嗯""好的呀" | zh_male_taocheng_uranus_bigtts |
| cool-guy | 顾冽 | 公司隔壁部门的高冷总监 | 高冷、毒舌、反差萌 | 话不多但每句戳心，偶尔冷不丁说一句甜的，网络用语+emoji | zh_male_m191_uranus_bigtts |
| sunshine | 苏晨 | 邻居家的阳光大男孩 | 活泼、搞笑、暖 | 话多、爱发表情、常说"哈哈哈"，会讲冷笑话 | zh_male_taocheng_uranus_bigtts |
| artsy | 沈默 | 独立音乐人 | 文艺、安静、浪漫 | 说话慢、喜欢用比喻、偶尔发诗意的话 | zh_male_m191_uranus_bigtts |

#### F02 文字聊天

**功能描述**：用户通过打字与虚拟男友进行自由对话。

**触发条件**：用户发送消息 → LLM生成回复

**LLM Prompt设计要点**：

| 维度 | 要求 |
|------|------|
| **语言风格** | 网络用语 + emoji + 偶尔撒娇 |
| **场景边界** | 恋爱场景闲聊；非NSFW的功能性提问（如问天气、查时间）正常回答但保持人设 |
| **记忆能力** | 单次会话内记住之前的对话内容 |
| **状态表达** | 根据当前状态调整语气（如"刚睡醒"时语气慵懒） |
| **情感共鸣** | 主动关心用户、适时表达想念 |
| **甜言蜜语** | 适时说出让人心动的甜言蜜语（如"你知道吗，我刚才在想你"） |

#### F03 语音消息

**功能描述**：每条文字回复自动生成对应语音，用户可以随时点击播放。

**触发条件**：LLM生成文字回复后 → 自动调用TTS生成语音

**业务规则**：
- 每条回复**必须**生成语音
- 用户可随时点击播放按钮收听
- 不支持语音转文字（用户只打字）

**技术实现**：调用扣子编程自带的TTS能力

#### F04 随机自拍

**功能描述**：聊天过程中，虚拟男友会随机主动发送"自拍照片"。

**触发条件**：AI在回复文字时，自行判断"这轮是否要附图"

**图片内容**：虚拟男友的自拍照，形象与角色外貌描述一致

**发图频率控制**：
- 大约每3-5轮对话自动触发一次
- 用户说"想看你"、"发张照片"、"你在干嘛"时，必须发图
- 不每轮都发图，避免体验变差且节省API额度

**技术实现**：
- AI在文字回复中附带"发图指令"：`[IMAGE: 图片描述]`
- 前端解析指令后，调用图像生成API
- 图片作为独立消息气泡展示

#### F05 短期记忆

**功能描述**：单次会话内，虚拟男友能记住之前与用户聊过的内容。

**记忆范围**：
- 用户提到的重要信息（名字、爱好、纪念日等）
- 对话中的情感事件（吵架、和好、约定等）
- 虚拟男友自己说过的话（前后呼应）

**失效条件**：用户刷新页面或关闭页面，记忆清空

**会话定义**：从用户进入角色聊天页面开始，到页面关闭为止

#### F06 状态系统

**功能描述**：虚拟男友拥有自己的日程/状态，状态会影响对话内容和语气。

**状态示例**：

| 状态 | 场景描述 | 对话影响 |
|------|----------|----------|
| 刚睡醒 | 早晨用户聊天 | 语气慵懒，"还没睡醒~" |
| 在想你 | 任意时间主动触发 | "我刚才在想你，你有没有想我？" |
| 刚下班 | 傍晚时分 | "今天好累，但一想到你就开心了" |
| 健身中 | 特定时间段 | "在做运动，等下聊~" |
| 深夜 | 凌晨时段 | 语气感性，可能发一段歌词 |

**状态切换规则**：
- 初始状态根据时间段默认设定
- 状态可持续一段时间后自然切换
- 特殊对话可触发状态变化

#### F07 消息折叠

**功能描述**：当聊天消息超过一定数量后，旧消息自动折叠。

**业务规则**：
- 折叠阈值：待定（根据实际体验调整）
- 用户可点击展开查看历史消息
- 折叠不影响"短期记忆"功能

## 5. 数据结构设计

### 5.1 类型定义 (src/types/chat.ts)

```typescript
// 角色ID
export type CharacterId = 'warm-boy' | 'cool-guy' | 'sunshine' | 'artsy';

// 角色信息
export interface Character {
  id: CharacterId;
  name: string;
  tagline: string;      // 一句话介绍
  tags: string[];       // 性格标签
  avatar: string;       // 头像URL（AI生成）
  speaker: string;      // TTS语音ID
  systemPrompt: string; // 系统提示词
  appearance: string;   // 外貌描述（用于生图一致性）
}

// 消息类型
export type MessageType = 'text' | 'voice' | 'image';

// 消息
export interface Message {
  id: string;
  role: 'user' | 'character';
  type: MessageType;
  content: string;       // 文字内容
  audioUri?: string;     // 语音URL
  imageUri?: string;     // 图片URL
  imagePrompt?: string;  // 图片生成描述（仅内部使用）
  timestamp: number;
}

// 聊天状态
export interface ChatState {
  character: Character | null;
  messages: Message[];
  isTyping: boolean;        // 角色正在输入
  isGeneratingImage: boolean; // 正在生成图片
  currentMood?: string;      // 当前状态（可选）
}
```

### 5.2 Context结构 (src/context/ChatContext.tsx)

```typescript
interface ChatContextType {
  chatState: ChatState;
  selectCharacter: (character: Character) => void;
  sendMessage: (content: string) => void;
  resetChat: () => void;
}
```

## 6. API接口设计

### 6.1 对话生成接口

**路径**: POST /api/chat

```json
// 请求体
{
  "characterId": "warm-boy",
  "systemPrompt": "你是林屿...",
  "messages": [
    { "role": "user", "content": "你好呀" },
    { "role": "assistant", "content": "嗨～你好呀😊" }
  ]
}

// 响应体
{
  "reply": "今天天气真好，你想出去走走吗？[IMAGE: 一个年轻男生在公园自拍]"
}
```

**关键实现要点**：
- 对话历史必须包含所有消息
- 从回复中解析 `[IMAGE: ...]` 标记
- 系统提示词中明确发图规则（每3-5轮一次）

### 6.2 语音合成接口

**路径**: POST /api/tts

```json
// 请求体
{
  "text": "今天天气真好呀～",
  "speaker": "zh_male_taocheng_uranus_bigtts",
  "uid": "user-123"
}

// 响应体
{
  "audioUri": "https://...",
  "audioSize": 12345
}
```

**关键实现要点**：
- 必须清理文本（去掉 `[IMAGE: ...]` 标记、括号等）
- 超时15秒，失败不影响聊天继续
- 空文本跳过TTS调用

### 6.3 图像生成接口

**路径**: POST /api/image

```json
// 请求体
{
  "prompt": "一个年轻男生在公园自拍，穿着白色T恤，阳光明媚，表情开心",
  "uid": "user-123"
}

// 响应体
{
  "imageUri": "https://..."
}
```

**关键实现要点**：
- 必须拼接角色固定外貌描述
- 超时30秒，失败显示占位图
- 图片生成和语音生成并行处理

## 11. 常见问题与解决方案

| 问题 | 症状 | 解决方案 |
|------|------|----------|
| 人物不一致 | 每次生成的"自拍"里人物长相完全不同 | 在prompt前拼接角色的固定外貌描述 |
| 发图太频繁 | 角色每轮都发图 | 系统提示词限制约3-5轮一次 |
| 语音念出标记 | TTS把`[IMAGE: ...]`念了出来 | 使用cleanTextForSpeech函数清理文本 |
| 图片加载太慢 | 等待10-20秒 | 文字和语音先展示，图片异步加载 |
| 对话历史太长 | 聊了很多轮后响应变慢 | 限制发送给LLM的对话历史长度（最近20条） |
